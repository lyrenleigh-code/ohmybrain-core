# Promote 工作流

把高价值对话结论写回 wiki，防止知识只停留在聊天记录里。

## 触发条件

以下情况必须执行 Promote：

- 对话中综合了多个来源，产生了 wiki 中不存在的新结论
- 纠正了 wiki 中的错误或过时信息
- 发现了之前未记录的概念间关联
- 做出了重要的技术决策或设计选择
- 用户明确要求 `/promote-answer`

## 步骤

### 1. 识别可沉淀的结论

从本次对话中提取：

| 类型 | 去向 | 示例 |
|------|------|------|
| 新概念定义 | `wiki/concepts/` 新建页 | 发现了一个新的技术术语或框架 |
| 对已有概念的补充 | `wiki/concepts/` 追加内容 | 为信道估计页补充了新的方法对比 |
| 新实体信息 | `wiki/entities/` 新建或更新 | 了解到一个新工具或人物 |
| 跨概念对比 | `wiki/comparisons/` 新建页 | A vs B 分析 |
| 系统性分析 | `wiki/explorations/` 新建页 | 深度研究笔记 |
| 专题综合 | `wiki/topics/` 新建或更新 | 跨概念的综合理解 |

### 2. 检查冲突

在写入前，读取目标页面当前内容：

- **无冲突**：直接追加新信息
- **有冲突**：不静默覆盖，使用以下格式标注：

```markdown
> [!warning] 信息更新
> 之前记载：[旧结论]
> 新发现（YYYY-MM-DD）：[新结论]
> 来源：[对话描述 / 资料引用]
```

### 3. 写入目标页面

**新建页面时：**
- 使用对应类型的模板（`.obsidian/templates/` 下）
- 必须包含 frontmatter（type、created、updated、tags）
- 必须包含 `[[wikilink]]` 交叉引用

**更新已有页面时：**
- 在相关章节末尾追加，不修改已有内容
- 注明来源和日期：`（YYYY-MM-DD 对话补充）`
- 更新 frontmatter 中的 `updated` 字段

### 4. 更新交叉引用

检查新写入的内容是否涉及其他 wiki 页面：
- 在其他相关页面中追加反向链接
- 确保 `[[wikilink]]` 双向连通

### 5. 更新 index 和 log

**wiki/index.md：**
- 新建页面时：在对应分类下追加条目
- 更新页面总数和最后更新日期

**wiki/log.md：**
```
## [YYYY-MM-DD] promote | 主题

从对话中沉淀结论到 wiki：
- 新建/更新：[页面名] — [简述内容]
- 涉及概念：[[concept1]], [[concept2]]
```

### 6. 运行 lint 验证

```bash
python3 scripts/lint_wiki.py
```

## 不应 Promote 的内容

- 纯调试过程和临时尝试
- 只对当前对话有意义的上下文
- 可以直接从代码或 git 历史中获取的信息
- 主观偏好（除非是明确的设计决策）

## 质量标准

- **可追溯**：每条结论都标注来源（对话日期、资料引用）
- **原子化**：一条结论写入一个最相关的页面，不在多个页面重复
- **可证伪**：结论表述要足够具体，未来能判断是否过时
- **有链接**：至少链接到一个相关概念页或实体页

## 示例

对话中发现："BEM 信道估计中，散布导频的设计比算法选择对性能的影响大 10-20dB。"

Promote 执行：
1. 目标页面：`wiki/concepts/channel-estimation-and-equalization.md`
2. 检查冲突：页面中已有 BEM 相关内容，无冲突
3. 在"关键技术"章节末尾追加：
   > BEM 信道估计中，散布导频的设计是精度的决定性因素，比算法选择对性能的影响大 10-20dB。（2026-04-12 对话补充，来源：[[uwacomm]] 模块 07 实验结论）
4. 更新 frontmatter 的 `updated` 字段
5. 更新 log
