# Spec 工作流

每个代码任务的起点。在动手之前，先把需求写清楚。

## 触发条件

- 新功能开发
- Bug 修复
- 重构任务
- 研究性实现（如算法验证）

## 步骤

### 1. 在 specs/active/ 创建 spec 文件

命名：`YYYY-MM-DD-任务名.md`

### 2. 填写 spec 模板

```markdown
# Task Spec: 任务标题

## 目标
一句话说明要做什么。

## 原因
为什么要做这个？解决什么问题？

## 范围
- 涉及哪些模块/文件
- 预期的变更量级

## 非目标
明确不做什么，避免范围蔓延。

## 依赖
- 需要哪些已有模块/接口
- 需要哪些外部资源

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
- [ ] 测试通过
- [ ] wiki 已同步（如需要）
```

### 3. 检查 wiki 上下文

在写 spec 时，查阅 wiki/ 中的相关页面：
- `wiki/architecture/` — 当前系统架构
- `wiki/modules/` — 已有模块接口
- `wiki/decisions/` — 历史设计决策
- `wiki/concepts/` — 相关技术概念

在 spec 中引用相关 wiki 页面，确保需求基于正确的系统理解。

### 4. 确认后进入 Plan 阶段

spec 完成后等待用户确认，然后执行 Plan 工作流（`workflows/plan.md`）。
