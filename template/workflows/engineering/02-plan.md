# Plan 工作流

将 spec 转化为可执行的实现计划。先拆解，再动手。

## 步骤

### 1. 读取 spec

从 `specs/active/` 读取目标 spec，理解目标、范围和验收标准。

### 2. 分析影响范围

| 检查项 | 方法 |
|--------|------|
| 影响的文件 | 搜索 src/、tests/ 中相关文件 |
| 依赖关系 | 检查模块间的调用链 |
| 已有实现 | 查阅 wiki/modules/ 确认是否有可复用的部分 |
| 历史决策 | 查阅 wiki/decisions/ 确认是否有相关约束 |

### 3. 在 plans/ 创建计划文件

命名：`YYYY-MM-DD-任务名.md`

### 4. 填写计划模板

```markdown
# Plan: 任务标题

## 目标
（复述 spec 中的目标）

## 非目标
（复述 spec 中的非目标）

## 影响的文件/模块

| 文件 | 变更类型 | 说明 |
|------|---------|------|
| src/xxx.py | 新建 | ... |
| tests/test_xxx.py | 新建 | ... |
| wiki/modules/xxx.md | 更新 | ... |

## 实现步骤

1. 步骤一
2. 步骤二
3. ...

## 测试策略

- 单元测试：...
- 集成测试：...
- 手动验证：...

## 风险和回退

| 风险 | 概率 | 应对 |
|------|------|------|
| ... | 低/中/高 | ... |

## 验证方式

- [ ] 测试命令：`pytest -q`
- [ ] lint：`python3 scripts/lint_wiki.py`
```

### 5. 确认后进入 Implement 阶段

计划完成后等待用户确认，然后执行 Implement 工作流（`workflows/implement.md`）。

## 约束

- 计划阶段不修改 src/ 或 tests/
- 如果分析过程中发现 spec 有问题，先更新 spec 再继续
