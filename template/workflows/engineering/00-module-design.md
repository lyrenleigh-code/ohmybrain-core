# 00 — 模块划分（可选工作流）

> **本工作流是可选的**——并非每个项目都需要模块化。先按下文"激活判断"决定是否引入，不符合条件直接跳到 [01-spec](01-spec.md)，保持平铺目录。
>
> 一旦激活，本项目后续所有 spec / plan / implement / validate 都按本规范处理模块边界。

## 1. 激活判断

### 跳过本工作流的场景

| 画像 | 典型 |
|------|------|
| 规模小 | < ~500 行 / < ~15 个函数 |
| 探索早期 | 算法结构还在试错，分界未稳定 |
| 单一职责工具 | 转录脚本、爬虫、数据迁移 |
| 一次性代码 | 不维护的分析、跑一次即弃 |

这些项目在根目录或 `src/` 下**平铺**即可，过早划模块反而增加导航成本。

### 激活本工作流的场景（任一命中）

- 函数数 > ~20 且职责多样
- 端到端有清晰阶段（信号链路 / 数据管道 / ETL 工序）
- 需要同类算法对比（强制统一接口以便横向比较）
- 长生命周期 + 多人协作（模块边界 = 责任边界）
- 已出现"这些文件不太搭"的感觉

激活时在项目 `CLAUDE.md` 顶部写一句 "本项目启用模块划分工作流（风格 A / B）"，标识语义。

## 2. 两种命名风格（项目内选一，不混用）

### 风格 A — 顺序编号（链路 / 管道类项目）

**适用**：端到端信号流、数据处理管道、按阶段推进的工序。

**形式**：`modules/NN_Name/`

**范例**（参考 UWAcomm 水声通信仿真，13 模块按信号链路编号）：

```
modules/
├── 01_SourceCoding/
├── 02_ChannelCoding/
├── ...
├── 07_ChannelEstEq/      # 复杂核心模块，规模允许远大于其它
├── ...
└── 13_SourceCode/
```

**优点**：编号反映数据流方向；依赖方向自然（下游依赖上游）；新成员看编号即懂脉络。

**约束**：编号不跳号；新增需插入合适位置并更新所有引用。

### 风格 B — 分类（算法对比 / 工具类项目）

**适用**：同一问题多种算法对比、工具性项目、无天然线性阶段。

**形式**：`{domain}/{category}/`

**范例**（参考 USBL 自定位）：

```
simulation/
├── config/     # 全局参数
├── core/       # 基础模块（阵列/信号/信道/坐标变换）
├── doa/        # DOA 估计算法（多种算法共享接口）
└── analysis/   # 分析工具
```

**优点**：按职责分类简洁；同类算法并列便于对比；扩展不破坏现有结构。

**约束**：分类名必须有明确职责边界；禁止 `utils/` `helpers/` `misc/` 这种**垃圾桶目录**。

## 3. 模块目录骨架

每个模块（风格 A 或 B）下：

```
<module>/
├── src/          # 源代码（多语言项目再分 Matlab/ / Python/ ...）
├── tests/        # 单元/集成测试（test_*.m / test_*.py）
├── README.md     # 模块接口与算法文档（章节沿用 03-implement §4）
└── (docs/)       # 可选：详细推导、论文引用、历史决策
```

README 必备章节见 [03-implement](03-implement.md) §4：简介 / 对外接口 / 内部接口 / 核心算法 / 使用示例 / 依赖关系 / 测试覆盖。

## 4. 统一接口约定

### `cfg` 作为第一参数

项目级全局参数通过一个 `cfg` 结构体传递，所有模块函数**第一参数**都是 `cfg`（或按需解构）——避免全局变量、避免散落的默认值。

```matlab
function y = module_func(cfg, x, extra_arg)
    fs = cfg.signal.fs;
    ...
end
```

### 同类算法共享签名

模块内有多个"同类算法"时（DOA / 信道估计 / 均衡器等），**所有实现必须共享调用签名**，便于 `run_comparison.m` 横向对比。

```matlab
% 同一模块内所有 DOA 算法统一接口
[theta_est, phi_est, P_spectrum] = doa_cbf  (z, array, cfg, grid);
[theta_est, phi_est, P_spectrum] = doa_mvdr (z, array, cfg, grid);
[theta_est, phi_est, P_spectrum] = doa_music(z, array, cfg, grid);
```

### 模块间通过函数调用

模块 B 用 A 的能力 → 导入并调用 A 的公开函数。**禁止**全局变量 / 隐式全局状态。

## 5. 依赖方向

| 层 | 依赖关系 | 例子 |
|----|---------|------|
| **横切** | 所有模块可依赖 | `config/` 、`core/` 基础工具 |
| **上游 → 下游** | 下游依赖上游，反之禁止 | UWAcomm 模块 12（迭代）可依赖 07（均衡器）；07 不能依赖 12 |
| **同层** | 原则上不互相依赖 | 同层模块通过共同调用 `core/` 实现能力共享 |

激活本工作流后，在 `wiki/architecture/` 画一张模块依赖图（文字或 Mermaid），作为接口审查基线。

## 6. 新增模块流程

1. **先写决策记录**：`wiki/decisions/YYYY-MM-DD-add-module-NN.md`
   - 为什么当前模块无法容纳？
   - 新模块与哪些模块依赖？
   - 统一签名如何对齐？
2. **更新目录登记**：项目 `CLAUDE.md` 目录结构 + `wiki/architecture/system-overview.md`
3. **建空骨架**：`src/` `tests/` `README.md`
4. **在首个 spec 里声明**：`## 涉及模块:` 字段标 `NN_Name (新建)`

## 7. 反例

- ❌ **模块多职责**：如"同步 + 帧组装"塞进同一模块——应拆
- ❌ **编号跳号**：01, 02, 04...（跳过 03）——除非 03 已删并有归档记录
- ❌ **风格混用**：同一项目里既有 `modules/07_Foo/` 又有 `src/algorithms/`
- ❌ **垃圾桶模块**：`utils/` `misc/` `helpers/`——每次往里塞前先问"能否归入已有职责模块"
- ❌ **接口漂移**：同类算法签名不一致（`doa_ml` 返回 4 值、`doa_cbf` 返回 3 值）
